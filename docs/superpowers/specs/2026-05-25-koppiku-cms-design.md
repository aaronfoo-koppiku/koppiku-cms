# Koppiku CMS — Design Spec
_2026-05-25_

## Overview

A cloud-based digital signage platform for Koppiku outlets. HQ marketing can upload content, build playlists, and schedule what plays on outlet TVs — all remotely. Screens at outlets run a browser-based player (Chrome on Android TV) that updates in real time when HQ publishes changes.

**Scale:** 100 outlets, 1–4 screens per outlet (~200 screens average)
**Timezone:** UTC+8 (Malaysia)
**Approach:** Build from scratch, Supabase-first, GCP for media

---

## Architecture

### Components

| Component | Technology | Hosting | Cost |
|---|---|---|---|
| CMS web app | Next.js 14 (App Router) | Vercel | Free tier |
| TV player app | React PWA | Vercel (separate deployment) | Free tier |
| Database + Auth + Realtime | Supabase (PostgreSQL) | Supabase managed | Free → $25/mo |
| Media storage | Google Cloud Storage | GCP credits | ~$0 to start |
| Media CDN | Cloud CDN | GCP credits | ~$0 to start |
| API logic | Supabase Edge Functions (Deno) | Supabase managed | Free (500k calls/mo) |

### Monorepo Structure

```
koppiku-cms/
  apps/
    cms/           ← Next.js admin panel
    tv-player/     ← React PWA fullscreen player
  packages/
    shared/        ← TypeScript types + Supabase client
  supabase/
    migrations/    ← SQL migrations
    functions/     ← Edge Functions
    seed.sql
  infra/           ← GCS bucket + CDN config
```

### Data Flow

```
HQ uploads media
  → Supabase Edge Function
  → GCS bucket (raw file)
  → CDN URL saved to media table in Supabase

HQ publishes playlist
  → Supabase DB update (schedules / playlists)
  → Supabase Realtime broadcasts to outlet:{id} channel
  → All TVs subscribed to that outlet receive update (<1s)
  → TV fetches new playlist, Service Worker pre-caches media

TV boots (first time)
  → Opens player.koppiku.com
  → Generates device UUID + 6-digit pairing code
  → Writes pending device row to Supabase
  → Displays pairing code fullscreen

HQ pairs device
  → Enters code in CMS → assigns outlet
  → Supabase Realtime notifies TV → status: active
  → TV subscribes to outlet channel, begins playback
```

## CMS Web App (Next.js)

### Pages

| Route | Purpose |
|---|---|
| `/login` | Supabase Auth email/password |
| `/dashboard` | Overview — device count, recent activity |
| `/media` | Media library — upload, preview, delete |
| `/playlists` | List playlists, create new |
| `/playlists/[id]` | Playlist builder — drag-and-drop items |
| `/schedules` | Assign playlists to outlets with time windows |
| `/outlets` | Manage outlets |
| `/devices` | Device list — pair, rename, see status |

### Media Upload Flow
1. User drags file onto `/media` upload zone
2. Browser sends file to Supabase Edge Function `/functions/v1/upload`
3. Edge Function streams file to GCS, generates signed CDN URL, creates thumbnail (images: sharp; videos: ffmpeg via Cloud Run job)
4. Metadata row inserted into `media` table
5. UI shows new item immediately (optimistic)

### Playlist Builder
- Drag-and-drop reorder via `@dnd-kit/core`
- Per-item duration override (images default: 10s; videos: use file duration)
- Publish button updates `status` → `published` and triggers Realtime broadcast

## TV Player App (React PWA)

### States

```
UNPAIRED  → shows 6-digit pairing code fullscreen
PAIRED    → fullscreen playback loop
OFFLINE   → continues from Service Worker cache, shows subtle indicator
```

### Playback Engine
- Fetch active schedule for current time (UTC+8) from Supabase on load
- Play `playlist_items` in `sequence` order, loop indefinitely
- Images: display for `display_duration_s` (default 10s)
- Videos: autoplay muted, loop=false, advance on `ended` event
- On Realtime update: finish current slide, then transition to new playlist
- No cursor, no browser chrome, no scroll bars

### Offline Resilience
- Service Worker (Workbox) caches all media CDN URLs for active playlist
- If Supabase unreachable: serve last-known playlist from `localStorage`
- On reconnect: Realtime re-subscribes, fetches diff, caches new media
- Heartbeat: POST to Edge Function every 30s to update `devices.last_seen`

### Android TV Setup
- Chrome browser opened in kiosk/fullscreen mode
- Bookmark `player.koppiku.com` as home page
- Device is paired once by outlet staff during setup

## Supabase Edge Functions

| Function | Trigger | Purpose |
|---|---|---|
| `upload` | POST (CMS) | Proxy file to GCS, return CDN URL |
| `pair-device` | POST (CMS) | Validate pairing code, link device to outlet |
| `heartbeat` | POST (TV player) | Update `devices.last_seen` |
| `resolve-schedule` | GET (TV player) | Return active playlist for outlet + current time |

## Security

- CMS routes protected by Supabase Auth session (JWT)
- Row-level security on all Supabase tables — users can only read/write their own org's data
- GCS bucket is private; CDN URLs are signed with 24h expiry
- TV player authenticates with a device-scoped anon key (read-only, outlet-scoped via RLS)
- Pairing codes expire after 10 minutes

## MVP Scope

### Phase 1 — Ship
- Media library (upload images + videos to GCS, preview, delete)
- Playlist builder (drag-and-drop, per-item duration, publish/unpublish)
- Basic scheduling (time window + day-of-week per outlet, priority for overlaps)
- Outlet + device management (create outlets, pair devices, online/offline status)
- TV player — fullscreen kiosk, Realtime sync, offline Service Worker
- Auth — Supabase email/password, single admin role

### Phase 2 — Post-MVP
- Role-based access control (super admin, HQ marketing, regional manager)
- Outlet grouping by region (push to "KL region" as a group)
- Scheduling calendar/timeline view
- Device screenshot preview in CMS
- Animated slide transitions (fade, slide, zoom)
- Playback analytics (impression logs, device uptime)

### Out of Scope
- POS integration
- AI features
- HTML widget slides
- Live tickers
- QR code overlays

## Key Development Notes

- All schedule time comparisons must use `Asia/Kuala_Lumpur` (UTC+8)
- TV player must prevent memory leaks during long-running sessions (release video object URLs, clear old cache entries)
- Videos must autoplay muted — Chrome on Android TV enforces this
- Service Worker must pre-cache next playlist before transition (no blank screens)
- CMS uses optimistic UI for all mutations — don't block on server round-trip
- Separate Vercel deployments for `cms` and `tv-player` — different domains, different CSP headers
- Design APIs with future analytics support: log playback events to a `playback_logs` table from day one (even if reporting UI is Phase 2)
